## ❌ Критический баг: даты импорта сдвинуты на -1 день

**Причина:** `src/data/parseFile.ts:86-94` — `fmtCell` использует `getUTC*()` вместо локальных геттеров. Для часовых поясов восточнее UTC (Москва UTC+3) Excel-дата читается на день раньше (08.07 → 07.07).

**Статус:** Не решено

**Что сделано:**

| Шаг | Что | Статус |
|---|---|---|
| 1 | `fmtCell` — `getUTC*()` → локальные геттеры | ✅ для новых импортов |
| 2 | Миграция v1: детекция `metricMax < logMax` | ❌ после переимпорта оба = July 8 |
| 3 | Миграция v1: удаление дубликатов по совпадению значений | ❌ переимпорт был до фикса, дубликатов нет |
| 4 | Миграция v1: сдвиг при охвате 2 месяцев + дата после последней метрики | ❌ `uploadedAt` в UTC, `metricMax` локально — сравнение ложно |

**Нужно:** Переделать детекцию сдвига в миграции v1 (`src/data/store.ts:763-...`). Возможное решение — сравнивать `uploadedAt` и `metricMax` в одной временной зоне, или добавить флаг при импорте о версии парсера.

## ❌ Баг фильтрации: группа + артикул

### Проблема 1 — фильтр по группе (склейке)
- В `AnalyticsTable.tsx:visible` (`matchGroup`) проверяется только `row.id === groupFilter`
- Дети группы (продукты с `row.parent === groupRowId`) **не проходят фильтр** и не отображаются в таблице
- `getTableData()` на бэкенде уже фильтрует по groupId, но `visible` дополнительно отсекает детей

**Частичный фикс:** добавлен `groupRowIds.has(row.parent)` — включены прямые дети, но не проверено для UNGROUPED_GROUP_ID

### Проблема 2 — фильтр по артикулу (SKU)
- Раньше был `<select>` со всеми артикулами — неудобно при сотнях SKU
- **Сделано:** заменён на `<input>` с выпадающим поиском (новая логика в `FilterBar.tsx`)
- **Не решено:** в `AnalyticsTable.tsx` логика `matchSku` проверяет точное совпадение `row.sku === skuFilter`, но при выборе артикула дети/родители не проходят фильтр — строка продукта может не отображаться, если она не совпадает по иерархии групп
- **useChartData.ts**: `skuFilter` добавлен и передан в графики — работает, но только если у продукта есть метрики в выбранном периоде

### Требуется
1. Переписать `visible` в `AnalyticsTable.tsx`: корректная фильтрация по иерархии (кабинет → группа → артикул)
2. Проверить поведение при комбинации filter + brand + group + sku
3. Протестировать UNGROUPED_GROUP_ID (товары без склейки) совместно с другими фильтрами

## 4. MVP — Воронка (FunnelPage)

**Бизнес-задача:** Понять, на каком этапе воронки теряются клиенты: Показы → Клики → Корзины → Заказы → Выкупы.

**Логика данных:**
- Источник: те же `DailyMetrics` (impressions, clicks, carts, orders, buyouts)
- Фильтр по кабинету/бренду/группе/артикулу (как на dashboard), период
- Разбивка по дням или агрегат за период

**MVP:**
- Таблица: этап → абсолютное значение → % конверсии от предыдущего этапа → % от показов
- Простая визуализация: горизонтальные бары или funnel SVG
- Сравнение двух периодов (период A vs B)

## 5. MVP — Точки входа (EntryPointsPage)

**Бизнес-задача:** Какие артикулы/SKU приносят основные показы, клики, заказы — точки входа в ассортимент.

**Логика данных:**
- Группировка метрик по SKU (продуктам) за период
- Сортировка по убыванию выбранной метрики (показы/заказы/выручка)

**MVP:**
- Таблица: SKU → показы → клики → корзины → заказы → выручка
- Сортировка по колонкам
- Top-N выборка (топ-10/20/50/все)
- Фильтр по кабинету/бренду/группе, период

## 6. MVP — Поисковые фразы (SearchPhrasesPage)

**Бизнес-задача:** Какие поисковые запросы приводят к показам/кликам/заказам. Оптимизация рекламных кампаний.

**Логика данных:**
- Данные из рекламного источника (xway) — требуется импорт с поисковыми фразами
- Если таких данных нет — заглушка "Импортируйте данные поисковых запросов"

**MVP:**
- Таблица: поисковая фраза → показы → клики → CTR → расходы (ad_spend) → заказы → DRR
- Сортировка, топ-N
- Фильтр по кабинету, период

## 7. Доработка — План (PlanningPage)

**Что есть сейчас:** Таблица с планами по кабинетам/группам/продуктам (заказы, сумма, прибыль). Редактирование ячеек, авто-суммирование детей в родителя.

**Финальный прототип:**
- Полностью рабочее редактирование (inline) всех полей плана
- Авто-пересчёт иерархии (изменение продукта → сумма группы → сумма кабинета)
- Импорт планов из шаблона (CSV/excel)
- Экспорт планов
- Визуализация: progress bars выполнения плана (факт vs план) — данные из DailyMetrics
- Фильтр по кабинету
- Выбор периода (месяц/квартал)

## 8. Доработка — Рентабельность (ProfitabilityPage)

**Что есть сейчас:** Таблица profitability с данными фактической прибыли/маржи/выручки. Редактирование расходов (эксплуатационные расходы).

**Финальный прототип:**
- Импорт данных фактической прибыли (CSV с SKU, период, profit, margin, revenue)
- Редактирование накладных расходов (операционные расходы) — влияют на итоговую чистую прибыль
- Расчёт рентабельности с учётом накладных расходов
- Фильтр по кабинету/бренду/группе, период
- История изменений расходов
- Сводка: выручка, себестоимость, валовая прибыль, накладные расходы, чистая прибыль, рентабельность
- Экспорт в CSV
